# 函数调用链与全局变量数据流分析工具 —— 需求规范文档
---

## 1. 引言

### 1.1 项目背景
大型操作系统项目（如Linux内核、OpenHarmony）的源码规模庞大，函数间调用关系错综复杂，全局变量的跨文件传播也常导致难以追踪的副作用。开发者、安全研究人员和架构师需要一种高效、交互式的工具来探索和理解这些静态结构。现有的工具要么无法处理千万级代码，要么缺乏可定制的可视化和增量查询能力。

本项目旨在开发一款基于Clang LibTooling的**VSCode插件**，能够解析Linux内核及OpenHarmony级别的源码，构建函数调用链和全局变量数据流图，并提供基于**El Grapho**的交互式图谱，帮助用户快速理解代码结构和数据流动。

### 1.2 讨论回顾
本需求文档的形成基于多轮深入的技术讨论，涵盖了以下核心主题：
- **可行性分析**：使用Clang LibTooling实现函数调用链解析的可行性，以及跨文件关联的挑战。
- **USR跨文件关联**：如何利用Clang的USR（统一符号解析）唯一标识函数，实现跨编译单元的函数关联。
- **按需构建**：避免全量解析的开销，采用“核心模块全量解析 + 其他模块懒加载”的策略，结合文件索引实现递归按需分析。
- **持久化存储**：为支持快速查询和复用，设计基于RocksDB的持久化存储方案，考虑节点、边、变量访问记录等数据结构，并支持增量更新。
- **处理头文件缺失**：针对非项目头文件缺失的问题，采用`--isystem`替换`-I`的策略，使Clang将外部头文件视为系统头，从而继续解析；并在回调中使用`isInSystemHeader()`过滤掉标准库和外部代码。
- **全局变量数据流图**：在调用图基础上扩展，追踪全局变量的定义、使用及跨函数传播，提出函数内数据流分析和跨函数传播的简化策略。
- **大规模可视化**：选择El Grapho作为前端渲染库，因其支持百万级节点交互；并提出后端数据分层、前端渐进渲染的架构。
- **VSCode集成**：最终工具以VSCode插件形式交付，提供搜索、图探索、路径高亮等交互功能。

本需求文档将这些讨论成果转化为明确的功能和非功能需求，并重点细化了**懒加载**和**全局变量数据流分析**的具体需求，同时明确了可视化呈现的范围限定为用户查询的特定函数调用链和特定全局变量数据流。

### 1.3 目标
- 构建一个VSCode插件，集成函数调用链与全局变量数据流分析的核心功能。
- 后端基于Clang LibTooling，能够解析Linux内核及OpenHarmony级别源码，并持久化存储分析结果。
- 前端使用El Grapho渲染图谱，仅呈现用户当前查询的**特定函数调用链**和**特定全局变量的数据流**，支持百万级节点交互，并提供丰富的查询和可视化定制能力。
- 支持增量更新和多版本代码库管理。

### 1.4 预期用户
- VSCode环境下的内核/系统开发者
- 安全研究人员
- 架构师/技术管理者

## 2. 总体描述

### 2.1 功能概述
工具由三部分组成：
- **VSCode插件前端**：提供用户交互界面，集成El Grapho图谱、搜索面板、详情展示等。图谱仅展示用户主动查询的函数调用链或全局变量数据流，而非全量图。
- **分析后端服务**（可选）：运行在本地或远程，提供REST API供插件调用，封装所有分析逻辑和持久化存储。
- **核心分析器**：基于Clang LibTooling，执行实际的源码解析和数据提取。

插件与后端的通信可以是本地的（通过进程间通信或本地HTTP服务），也可以是远程的（适用于团队共享分析结果）。默认采用本地模式，所有数据存储在用户本地。

### 2.2 假设与约束
- 用户已安装Clang并配置好编译环境。
- 项目必须提供`compile_commands.json`编译数据库。
- 源码规模可达千万行，节点数50万+，边数千万级。
- 插件需兼容VSCode 1.70以上版本。
- 前端需在主流浏览器（VSCode内嵌的Electron环境）上流畅运行。

## 3. 功能需求

### 3.1 插件核心功能

#### 3.1.1 项目配置
- 用户可通过插件设置指定项目根目录、编译数据库路径。
- 支持配置**核心模块列表**（如`kernel/`、`mm/`），这些模块将被全量解析并常驻内存。
- 允许用户自定义编译参数转换规则（如自动将`-I`替换为`--isystem`以忽略外部头文件）。

#### 3.1.2 解析触发
- 提供“开始解析”命令，触发后端分析器遍历所有源文件（或仅核心模块）。
- 解析过程可暂停/恢复，支持断点续传。
- 解析进度通过VSCode状态栏或输出通道实时显示。

#### 3.1.3 函数搜索与跳转
- 支持按函数名模糊搜索，结果列表显示函数所在文件和模块。
- 点击搜索结果可直接跳转到源码位置（VSCode打开对应文件并定位到行）。
- 搜索支持过滤（按模块、文件类型）。

#### 3.1.4 调用链图谱（按需呈现）
- 当用户选择一个函数作为根节点时，插件向后端请求该函数的调用链子图（正向/反向，可指定深度）。
- 图谱仅展示从该根节点出发的调用链（正向或反向），不展示全量调用图。
- 支持在现有图谱基础上**展开**：点击某个已显示的节点，请求该节点的直接调用者/被调用者，动态添加到当前视图中。
- 支持设置展开深度限制，避免一次性加载过多节点。
- 提供路径查询：输入两个函数，请求并高亮它们之间的所有调用路径（作为子图呈现）。

#### 3.1.5 全局变量数据流图（按需呈现）
- 支持按变量名搜索全局变量，显示其定义位置和类型。
- 当用户选择一个全局变量时，插件请求该变量的数据流图：
  - 默认展示直接读写该变量的函数节点，以及它们之间的调用边（体现传播路径）。
  - 可进一步展开，查看更深层的传播（如通过调用链传递）。
- 数据流图以变量为中心，只包含与该变量相关的函数节点和边，不包含无关函数。
- 支持过滤：只显示写操作、只显示读操作，或两者叠加。

#### 3.1.6 可视化交互
- 图谱基于El Grapho渲染，支持：
  - 缩放、平移、旋转
  - 节点悬停显示详细信息（函数名、文件、行号）
  - 节点点击打开源码或展开邻居（懒加载）
  - 边悬停显示调用点信息或数据流类型
- 提供多种布局切换（力导向、层次、聚类）以适应不同查询场景。
- 支持导出当前视图为PNG/SVG或JSON数据。

#### 3.1.7 多版本管理
- 支持在同一工作区管理多个代码版本（如不同内核版本）。
- 每个版本的分析数据独立存储，用户可随时切换。

### 3.2 后端分析功能

#### 3.2.1 编译环境适配
- 读取`compile_commands.json`，允许用户通过插件设置修改参数（如将`-I`替换为`--isystem`）。
- 支持通过`--isystem`标记外部头文件，使Clang在缺失时继续解析，并在分析结果中将这些节点标记为“系统”，便于前端过滤。

#### 3.2.2 函数调用链提取
- 基于AST匹配器，提取所有函数定义和调用关系。
- 生成USR作为函数唯一标识，并记录函数名、文件、行号、签名、所属模块。
- 存储正向和反向调用边（caller → callee），并附带调用点位置。

#### 3.2.3 全局变量访问提取
- 识别所有具有全局存储类别的变量（`VarDecl` with `hasGlobalStorage`）。
- 在每个函数内，匹配对全局变量的引用（`DeclRefExpr`），判断是读还是写（通过是否为赋值左值）。
- 记录每个函数对每个全局变量的访问列表（读/写、位置）。

### 3.3 懒加载需求（详细说明）

#### 3.3.1 核心模块全量解析
- 用户可在配置中指定一组**核心模块**（如`kernel/`、`mm/`、`fs/`等）。这些模块的**所有**源文件将在初始解析时被完整分析。
- 核心模块的调用关系和数据流信息在解析后**常驻内存**（可配置，如永不过期），以确保对这些模块的查询达到最快响应。
- 核心模块的解析结果同时持久化到RocksDB，用于备份和恢复。

#### 3.3.2 非核心模块懒加载
- 对于非核心模块的函数，不预先解析，仅在查询涉及时才触发解析。
- **触发条件**：
  - 用户在调用链图谱中点击一个尚未展开的节点（函数）。
  - 用户搜索一个函数并选择查看其详情。
  - 用户查询一个全局变量的访问列表，涉及某个未解析的函数。
- **文件定位**：
  - 当需要解析一个函数时，首先通过该函数的USR（若已知）在RocksDB中查找其定义文件。若尚未记录，则通过预先构建的**文件名→函数索引**（可选）或通过函数名猜测（结合项目结构）定位文件。
  - 定位到文件后，检查该文件是否已被解析。若未解析，则将其加入待解析队列。
- **模块级解析**：
  - 懒加载采用**模块级粒度**：当需要解析一个文件时，通常解析该文件所属的整个模块（由用户配置定义，如按目录划分）。这样做可以避免后续重复解析同一模块的其他文件。
  - 解析模块时，使用与核心模块相同的流程：遍历模块内所有源文件，提取函数和调用信息，并将结果写入RocksDB。
- **内存缓存**：
  - 解析后的非核心模块数据（函数节点和边）会加入内存缓存，但受**LRU（最近最少使用）策略**管理。
  - 缓存大小可配置（如最大5000个函数），当缓存满时淘汰最久未使用的函数及其调用边（但保留其基本元数据，以便快速重新加载）。
  - 被淘汰的函数若再次被查询，需重新从RocksDB加载（可能触发再次解析，但数据已持久化，无需重新编译）。
- **并发与队列**：
  - 懒加载解析任务放入队列，由后台线程处理，避免阻塞UI。
  - 相同模块的解析请求合并，避免重复解析。

#### 3.3.3 断点续传与增量更新
- 解析过程中若中断（如用户取消、进程崩溃），下次启动时可从上次解析的最后一个文件继续，无需重新开始。
- 当源码更新时，插件可检测修改的文件，并仅重新解析这些文件所属的模块（增量更新），更新RocksDB中的数据，并刷新内存缓存（可简单清除受影响模块的缓存，或局部更新）。

### 3.4 全局变量数据流分析需求（详细说明）

#### 3.4.1 数据流分析的范围与粒度
- **分析对象**：所有具有全局存储类别的变量（包括文件作用域的`static`变量、全局变量、`extern`变量）。
- **基本单元**：以函数为基本分析单元，记录每个函数内对全局变量的**读（R）**和**写（W）**操作。
- **操作位置**：记录每个访问点所在的文件名、行号、列号，以及该访问是在哪个函数内发生的。

#### 3.4.2 函数内数据流（第一阶段简化）
- 在第一阶段，不进行复杂的函数内数据流分析（如基本块间的定义-使用链），仅记录**每个函数内部**对全局变量的所有读写点。
- 但为了支持后续跨函数传播，需计算每个函数的**副作用摘要**：
  - `modifies`集合：该函数可能修改的全局变量列表（通过直接赋值或调用其他可能修改的函数）。
  - `references`集合：该函数读取的全局变量列表。
- 副作用摘要通过结合函数内直接访问和调用图传播计算得出（见下节）。

#### 3.4.3 跨函数数据流传播
- **传播规则**：
  - 如果函数 `A` 调用了函数 `B`，且 `B` 可能修改全局变量 `g`，那么 `A` 也可能修改 `g`（传递闭包）。
  - 如果函数 `A` 读取了全局变量 `g`，且 `g` 的值可能在调用 `A` 之前被其他函数修改，则 `g` 的“定义-使用”路径需包含这些调用者。
- **数据流图的构建**：
  - **节点**：函数、全局变量（作为特殊节点）。
  - **边类型**：
    - 调用边（已有）
    - 直接访问边：函数 → 变量（标记为读/写）
    - 间接访问边：通过调用链传递的变量修改（函数 → 函数 → 变量）
  - 数据流图是**有向图**，方向为“定义（写）流向使用（读）”。
- **查询类型**：
  1. **变量使用者查询**：给定全局变量，找出所有读取该变量的函数（直接或间接）。
  2. **变量修改者查询**：找出所有可能修改该变量的函数。
  3. **变量传播路径查询**：给定一个变量和一个起始函数，找出从起始函数出发，经过哪些函数最终读取/修改该变量的路径（类似调用链，但关注变量）。
  4. **两点间数据流**：给定两个函数，找出它们之间传递特定变量的路径。

#### 3.4.4 实现策略（简化）
- **直接访问收集**：在解析每个函数时，收集其对全局变量的直接读写，存入RocksDB（`access:func:<func_usr>:<var_usr>`）。
- **副作用传播**：
  - 在函数调用图构建完成后，对每个函数计算其修改集（`modifies`）和引用集（`references`）。这是一个不动点迭代过程：
    1. 初始化：函数的修改集 = 直接写访问的变量；引用集 = 直接读访问的变量。
    2. 迭代：若函数A调用函数B，则将B的修改集并入A的修改集，将B的引用集并入A的引用集。
    3. 重复直到稳定（由于调用图可能有环，需使用工作列表算法）。
  - 计算结果可持久化，用于加速数据流查询。
- **数据流路径生成**：
  - 对于变量 `g`，从起始函数出发，进行带上下文的搜索：
    - 若当前函数直接写 `g`，记录路径。
    - 若当前函数调用其他函数，且被调用者可能写 `g`，则递归搜索被调用者。
  - 为避免路径爆炸，可限制深度、只显示直接路径、或采用聚合展示。

#### 3.4.5 可视化呈现（按需）
- 在El Grapho图谱中，全局变量作为特殊节点（如菱形图标）展示。
- 数据流边用不同颜色（如蓝色）区别于调用边（灰色）。
- 用户切换至“数据流视图”时，图谱仅显示与当前查询的全局变量相关的节点和边，不显示无关函数。
- 支持过滤：只显示对某个变量的读写路径。

### 3.5 过滤机制（确保只分析项目代码）

- **编译阶段过滤**：通过将外部头文件路径设为`--isystem`，使Clang能够解析但标记为系统头，从而在AST中可以通过`SourceManager::isInSystemHeader()`识别。
- **回调阶段过滤**：在所有匹配回调中，使用`isInSystemHeader()`或`isExpansionInSystemHeader()`跳过系统头文件中的节点，确保只有项目代码被记录。
- **标准库过滤**：由于标准库路径通常也在系统头中，因此自动被排除。用户也可额外指定过滤路径。

## 4. 非功能需求

### 4.1 性能指标
- **后端解析**：全量解析Linux内核（约3000万行，50万函数）应在30分钟内完成（16核并行）。
- **查询响应**：
  - 函数搜索：< 200ms
  - 加载函数直接邻居：< 200ms（从RocksDB读取）
  - 懒加载触发新模块解析：用户可接受秒级延迟（如3-5秒），但应有进度提示。
  - 数据流路径查询（简单路径）：< 1s
- **前端渲染**：
  - 加载5000节点/10000边：< 2秒完成首次渲染（针对单个查询子图）
  - 交互（缩放、平移）：60 fps
  - 节点展开加载新数据：< 500ms（从请求到渲染，若数据已缓存）
- **内存占用**：
  - 后端服务：核心模块常驻内存不超过4GB，LRU缓存限制总内存8GB以内
  - 插件前端：单个标签页内存占用 < 500MB

### 4.2 可扩展性
- 支持添加新模块或新版本代码，无需全量重建。
- 支持分布式部署：当单机无法容纳全量数据时，可按模块分片存储到多台RocksDB实例，查询时通过路由层分发。

### 4.3 可靠性
- RocksDB数据定期校验，防止损坏。
- 解析过程中断可恢复：记录已解析文件列表，下次从中断处继续。
- 插件异常时能恢复视图状态（如保存当前图谱布局）。

### 4.4 易用性
- 插件提供详细的配置向导，帮助用户快速设置项目。
- 图谱操作符合直觉，提供快捷键和工具提示。
- 提供示例项目和预解析数据供新用户试用。

## 5. 技术架构

### 5.1 整体架构图
```
+-----------------------------------+
|         VSCode 插件前端           |
|  (TypeScript + React + El Grapho) |
+------------------^----------------+
                   | REST API (HTTP)
+------------------v----------------+
|       本地分析后端服务             |
|  (Node.js 封装，调用 Clang 工具)   |
+------------------^----------------+
                   | 进程间通信/直接调用
+------------------v----------------+
|       核心分析器 (C++)            |
|  (Clang LibTooling + RocksDB)     |
+-----------------------------------+
```

### 5.2 模块划分

#### 5.2.1 VSCode插件前端
- **扩展激活**：监听VSCode启动，注册命令、视图、配置。
- **Webview面板**：嵌入El Grapho图谱和交互UI，通过`postMessage`与扩展主进程通信。
- **API客户端**：封装对本地后端服务的HTTP请求，处理数据缓存和错误重试。
- **配置管理**：读取和更新VSCode设置，传递给后端。

#### 5.2.2 本地后端服务
- 使用Node.js编写，负责启动和管理核心分析器进程，提供REST API。
- 内部维护RocksDB连接池，处理并发请求。
- 可选的构建任务队列：当用户触发解析时，将任务加入队列，避免阻塞查询。

#### 5.2.3 核心分析器（C++）
- 基于Clang LibTooling，使用ASTMatcher提取信息。
- 集成RocksDB进行数据持久化。
- 提供C API或通过命令行交互，由Node.js服务调用。

### 5.3 数据格式（Protobuf定义）

```protobuf
syntax = "proto3";

message FunctionInfo {
  string usr = 1;
  string name = 2;
  string file = 3;
  int32 line = 4;
  int32 column = 5;
  string module = 6;
  string signature = 7;
  repeated string modifies = 8;  // 可能修改的全局变量USR列表（副作用摘要）

}

  string name = 2;
  string file = 3;
  int32 line = 4;
  string type = 5;
  bool is_extern = 6;

message GlobalVarAccess {
  string var_usr = 1;
  string function_usr = 2;

  string access_file = 4;
  int32 access_line = 5;
}

message DataFlowEdge {
  string from_usr = 1;      // 源节点（函数或变量）
  string to_usr = 2;        // 目标节点（函数或变量）
  string var_usr = 3;       // 涉及的全局变量
  string type = 4;          // "call", "direct_read", "direct_write", "propagated"
```

## 6. 接口定义（REST API）

| 端点 | 方法 | 参数 | 返回 |
|------|------|------|------|
| `/api/search/functions` | GET | `q`: 模糊查询 | `[{usr, name, file, module}]` |
| `/api/functions/<usr>` | GET | `include_neighbors=true` (可选) | `FunctionInfo` + `callers`/`callees` (前50) |
| `/api/callgraph/forward/<usr>` | GET | `depth` (默认2) | 返回以`<usr>`为起点的正向调用子图（节点和边），仅包含指定深度内的函数 |
| `/api/callgraph/backward/<usr>` | GET | `depth` | 返回反向调用子图 |
| `/api/search/variables` | GET | `q` | `[{usr, name, file, module}]` |
| `/api/variables/<usr>` | GET | - | `GlobalVarInfo` |
| `/api/variables/<usr>/accesses` | GET | `function_usr` (可选过滤) | `[{function_usr, is_write, location}]` |
| `/api/dataflow/variable/<usr>` | GET | `from_usr` (可选), `to_usr` (可选), `depth` (默认3) | 返回以变量`<usr>`为中心的数据流子图，包含相关函数节点和数据流边 |

## 7. 可视化定制（El Grapho）

### 7.1 集成方式
- 在VSCode Webview中引入El Grapho库（通过CDN或本地打包）。
- 初始化`ElGrapho`实例，绑定到canvas容器，监听消息事件。
- 数据格式适配：El Grapho接受`{nodes: [{id, label, group, size, ...}], edges: [{from, to}]}`。我们将USR映射为`id`，函数名作为`label`，模块作为`group`用于着色。

### 7.2 按需呈现原则
- **绝不加载全量图**：图谱始终基于用户当前查询构建子图，仅包含查询结果中的节点和边。
- **初始查询**：用户搜索函数或变量后，点击查看调用链/数据流，插件请求对应API，获得子图数据后渲染。
- **增量扩展**：在已渲染的子图上，用户可点击节点展开其邻居，此时插件请求该节点的直接调用者/被调用者，将新节点和边合并到现有图中。
- **视图切换**：当用户切换到另一个函数或变量的查询时，清空当前图谱，重新渲染新子图。

### 7.3 定制需求
- **节点着色**：
  - 核心模块函数用深色，非核心模块用浅色。
  - 当前搜索或选中的节点高亮（外框或闪烁）。
  - 全局变量节点使用特殊形状（如菱形）或颜色区分。
- **边着色**：
  - 调用边默认灰色，数据流边（变量传播）用蓝色。
  - 可根据调用次数调整边粗细。
- **交互增强**：
  - 右键菜单：展开、收起、设为根、查看源码。
  - 键盘导航：方向键移动视图，回车展开选中节点。
  - 迷你地图：概览当前子图，可拖动视口。
- **增量加载**：
  - 当用户点击节点“加载更多”时，前端发送请求获取该节点的直接邻居，并合并到现有图中。
  - El Grapho提供`addNodes`和`addEdges`方法实现动态更新。

### 7.4 性能优化
- **数据压缩**：API返回的节点和边使用紧凑格式（如仅包含必要字段），减少传输量。
- **Web Worker布局**：将El Grapho的布局计算移至Worker线程，避免阻塞UI渲染。
- **细节层次**：在缩放级别低时，将同一模块的多个节点聚合成一个“簇”节点，减少绘制数量。放大后再展开。
- **虚拟化**：只渲染视口内的节点和边，配合El Grapho内置的culling机制。
